---
description: One-time project context gathering — personalizes all subsequent Zephr commands for your app
---

Ask the user the questions below, then write the answers to `ZEPHR-PROJECT.md` in the project root. Once this file exists, every other Zephr command reads it automatically to personalize output.

Run this once per project, at the start of a session on a new codebase.

---

**Questions to ask**

Ask these one at a time, or present them as a form. Wait for responses before generating the context file.

1. **App domain and purpose**
   "What does this app do, and who uses it?"
   *Example answers: "B2B SaaS for HR teams to manage payroll", "Consumer mobile app for tracking workouts", "Internal admin dashboard for a logistics company"*

2. **Primary user action**
   "What is the single most important thing a user comes to this app to do?"
   *Example: "Submit expense reports", "Review and approve loan applications", "Schedule social media posts"*

3. **Key screens**
   "List the 3–5 most important screens in the app."
   *Example: "Dashboard, Create Report, Report Detail, Approvals, Settings"*

4. **Brand tone**
   "How should the UI feel? Pick 2–3 words."
   *Examples: formal / trustworthy / enterprise; playful / friendly / approachable; minimal / focused / calm; bold / modern / fast*

5. **Existing design patterns to preserve**
   "Are there any specific UI patterns, colors, or conventions already in place that I should not change?"
   *Example: "We use a left sidebar nav, our brand color is #5B4CF5, tables always have row-level actions in a ••• menu"*

6. **Current tech stack**
   "What are you using? (React version, state management, CSS approach)"
   *Example: "React 18, Zustand, Tailwind + Zephr tokens, TypeScript"*

7. **What's most broken right now**
   "What's the #1 visual or UX problem you want fixed?"
   *Example: "The dashboard is overwhelming — too many cards above the fold", "The form validation is confusing — errors don't appear inline"*

---

**Write the context file**

After collecting answers, write `ZEPHR-PROJECT.md` with this structure:

```markdown
# Zephr Project Context

## App
[One paragraph: what the app does, who uses it, the primary action]

## Primary user task
[The single most important action, verbatim or paraphrased from the user]

## Key screens
[Bulleted list of 3–5 screens]

## Brand tone
[2–3 tone words + one sentence describing the visual register]

## Design constraints
[Existing patterns, colors, or conventions to preserve — or "None specified"]

## Tech stack
[Framework, state management, CSS approach]

## Top priority fix
[The #1 problem the user named]

## Zephr command context
Generated by /teach-zephr on [DATE].
All Zephr commands will use this context to personalize output.
```

---

**How subsequent commands use this file**

Once `ZEPHR-PROJECT.md` exists:

- `/clarify` will match copy rewrites to the brand tone
- `/scaffold [type]` will name sections and labels using the app domain
- `/bolder` and `/focus` will know which action is genuinely primary in this product
- `/critique` will evaluate the screen against the primary user task
- `/harden` will write error messages specific to this app's workflows
- `/colorize` will know whether semantic green = success or a brand decision

---

**Output**

Write `ZEPHR-PROJECT.md` to the project root. Confirm the file was written and show the user a summary of what was captured. Then suggest the next command to run based on what they said the top priority fix is:

- Layout/density problem → suggest `/distill` or `/critique`
- Missing states → suggest `/harden`
- Visual hierarchy → suggest `/bolder` or `/focus`
- Copy problems → suggest `/clarify`
- Starting fresh → suggest `/scaffold [page-type]`
